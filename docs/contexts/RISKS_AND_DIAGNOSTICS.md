# Haan 리스크 및 진단 방법

이 문서는 Haan 프로젝트의 핵심 가설(멀티턴 데이터 없이 멀티턴 능력 창발)이 **실패할 수 있는 구체적 메커니즘**과, 각각을 조기에 감지·구분하기 위한 진단 방법을 정리한다. 완전 실패 위험이 성공 가능성보다 크다고 보아야 하며, 이 문서의 목적은 "실패하더라도 왜 실패했는지 설명 가능한 상태"를 만드는 것이다.

## 1. 언어-태스크 스퓨리어스 상관관계 (가장 심각한 위험)

**메커니즘**: 학습 데이터 구조상 "영어=멀티턴, 한국어=싱글턴"이라는 상관관계가 완벽하게 존재한다. 모델이 turn-taking 능력을 언어-독립적인 구조적 패턴으로 학습하는 대신, "이 발화가 한국어 음향 토큰이면 한 턴 말하고 멈춰라"는 **지름길(shortcut)**을 학습할 위험이 크다. 이렇게 되면 한국어 입력에서는 사용자가 아무리 오래 말하거나 끼어들어도 모델이 계속 반이중처럼 행동할 수 있다. 일반적인 딥러닝의 shortcut learning 문제와 동일한 패턴.

**대응**:
- 배치 구성 시 언어와 태스크(멀티턴 vs 싱글턴)의 상관관계 완화 — 영어 데이터 일부를 싱글턴처럼 잘라 섞기 (스켈레톤 이식(`DATA_STRATEGY.md` §4.3)은 보류/비채택이나, 이 de-correlation 배치 구성 자체는 독립적으로 유효)
- Turn-taking 관련 내부 표현이 언어에 무관하게 활성화되는지 causal probing으로 직접 측정
- Phase 1~2에서 LoRA가 아닌 Full FT 사용 — 저랭크 어댑터는 shortcut 쪽으로 더 쉽게 수렴하는 경향 (`TRAINING_CURRICULUM.md` §3.2)

## 2. Joint Loss에서 그라디언트 지배

**메커니즘**: 싱글턴 TTS loss가 KD loss보다 최적화하기 쉬우면(더 빨리 낮은 loss에 도달), 학습이 진행될수록 모델이 TTS 목적함수 쪽으로 편향되며 멀티턴 관련 표현이 서서히 지워질 수 있다(catastrophic forgetting과 유사). 데이터 비율이나 loss weight를 잘못 잡으면 영어 멀티턴 능력 자체도 퇴화해, 한국어로 전이될 게 남아있지 않게 될 수 있다.

**대응**:
- 두 태스크의 loss scale/gradient norm을 학습 내내 모니터링
- 영어 held-out 멀티턴 성능을 지속 추적 — 한국어 투입 이후 급격히 떨어지는지 관찰
- PCGrad류 gradient surgery 기법 고려, 최소한 loss weighting ablation 리포트

## 3. "듣기+말하기 동시성" 채널이 한국어 조건에서 학습되지 않음

**메커니즘**: 싱글턴 TTS 데이터는 보통 화자가 혼자 말하는 클린 오디오라, 입력 스트림을 계속 청취하며 처리하는 채널이 한국어 조건에서 한 번도 활성화된 채로 학습되지 않을 수 있다. Full-duplex의 핵심은 "말하면서 동시에 듣는" 두 스트림의 병렬 처리인데, 이를 언어와 무관하게 일반화된 메커니즘으로 학습했는지, 아니면 영어 데이터의 특정 조건에서만 발동하는 조건부 회로로 학습했는지는 실제로 열어보기 전까진 알 수 없다.

**아키텍처 연결**: 원 Moshi의 self/user 완전 분리 임베딩 구조(`ARCHITECTURE.md` §2)에서는 이 위험이 특히 크다 — 유저 채널 임베딩(인덱스 8~15)이 한국어 콘텐츠 그라디언트를 아예 받지 못하기 때문. 본 트랙이 채택한 공유 임베딩+Role Token 설계(`ARCHITECTURE.md` §3)에서는 임베딩 표현 자체는 공유되어 완화되지만, 그 표현을 유저 role로 "읽어내는" 어텐션 회로가 한국어 콘텐츠로 노출된 적이 없다는 문제는 완전히 사라지지 않는다.

**대응**: `DATA_STRATEGY.md` §4.2의 ASR 방향 양방향 재활용을 최소한의 노출 경로로 확보. 학습 후 self.emb(혹은 공유 임베딩 사용 시 해당 파라미터)가 초기화값 대비 얼마나 이동했는지(L2 거리/코사인 유사도)를 직접 측정해 정량 진단 가능.

## 4. 메커니즘(타이밍) 전이는 되지만 내용(일관성) 전이는 안 되는 부분적 실패

**메커니즘**: 침묵·맞장구·끼어들기 타이밍 같은 "형식"은 전이되더라도, 여러 턴에 걸쳐 논리적으로 일관된 한국어 대화 내용을 생성하는 "의미적 멀티턴 능력"은 다른 문제다. Qwen3 백본이 텍스트 레벨에서는 이미 멀티턴 한국어 대화 능력을 갖고 있어도, 그게 오디오 생성 헤드까지 그대로 내려오는지는 별도 검증이 필요하다.

**대응**: "turn-taking 메커니즘 전이"와 "대화 내용 일관성 전이"를 **반드시 분리해서 평가** — 그렇지 않으면 실패인지 성공인지조차 판단하기 어려움. 평가 시 ASR 기반 WER/CER로 내용 정확성을, 별도로 타이밍/오버랩 패턴 분석으로 메커니즘 전이를 각각 측정.

## 5. RVQ 코드북 초기화가 오히려 방해가 될 가능성

**메커니즘**: Moshi 코드북 값으로 초기화한 임베딩이 영어 음성 분포에 강하게 편향된 지역(local optimum)으로 학습을 유도해, 한국어 음향 특성에 맞는 표현으로 이동하는 데 오히려 더 많은 스텝이 걸리거나 그 근방에 갇힐 수 있다.

**대응**: Random init 대비 ablation으로 실제 효과 검증 (`TRAINING_CURRICULUM.md` §2 Phase 5).

## 6. Mimi 코덱 레벨의 병목 (코덱 vs LM 전이의 구분)

**메커니즘**: WavLM(Mimi 시멘틱 토큰의 증류 소스)이 영어 중심으로 학습되어, 코덱 자체가 한국어/일본어 음운(피치 악센트, 모라 리듬 등)을 표현할 어휘를 충분히 갖지 못할 가능성. 이 경우 아무리 백본이 완벽히 전이해도 코덱이 표현 못 하는 음운적 차이는 생성될 수 없다.

**대응**: `ARCHITECTURE.md` §4.3의 Mimi round-trip 재구성 테스트를 본 학습 전에 선행해, "코덱 병목" vs "LM 전이 병목"을 사전에 분리 진단. J-Moshi의 사전 실험(Mimi 동결 후 일본어 재구성 확인)이 이 우려를 상당 부분 완화하는 선례이나, 한국어에 대해서도 별도 재현 필요.

## 7. KD(Knowledge Distillation) 관련 주의점

> **채택 요약 (`ARCHITECTURE.md` §5.1)**: KD는 **Mimi semantic 토큰(level-0) logit**에 한정한다. acoustic(1~7)은 기본 KD 제외(음색은 voice-prompt), 단계적 graft만 별도(Phase 3.5). hidden-state KD·텍스트 logit KD는 폐기(§7.1·§7.2). KD의 실질 목적은 turn-taking 동시성 행동 전이.

### 7.1 백본 아키텍처 불일치로 인한 KD 레벨 제한
Helium(원본 Moshi)과 Qwen3 8B는 hidden dim, layer 수, attention 구조가 다르므로 intermediate feature-level KD(hidden state matching)는 사실상 불가능 → **hidden-state KD 폐기**. KD 가능 지점은 오디오 RVQ 토큰 출력 로짓(같은 Mimi 코덱 공유)과 텍스트 스트림으로 제한되며, 그중 **semantic 토큰(level-0) logit KD가 주 경로**다(공유 코덱이라 출력공간 동일 → projection 불필요).

### 7.2 텍스트 스트림 토크나이저 불일치
Helium과 Qwen3의 텍스트 토크나이저가 다르므로 내적 독백(Inner Monologue) 텍스트 토큰에 대해서는 로짓 단위 KD가 불가능. 티처 생성 텍스트를 디코딩 → 리토크나이즈해서 student 타겟으로 쓰는 시퀀스 레벨 방식(SeqKD)이 필요하며, 이 경우 소프트 타겟의 정보량(분포 전체)이 사라지고 하드 라벨만 남는 손실이 있음.

### 7.3 온폴리시 vs 오프폴리시 문제 (Exposure Bias)
Full-duplex 멀티턴 대화는 긴 자기회귀 시퀀스. 티처의 정답 궤적(teacher forcing)만으로 KD하면, student가 스스로 생성한 궤적에서 조금씩 벗어나는 순간 이후 전체 대화가 티처가 한 번도 보지 못한 분포로 흘러가며 오류가 누적된다(long-horizon drift). 멀티턴에서 특히 치명적. **순수 teacher-forcing KD보다는 student가 일부 자기 생성한 롤아웃에 대해 티처가 스코어링하는 on-policy 방식(혹은 최소한 scheduled sampling)을 섞는 것이 안전.**

**채택 절충안 (오프라인 기본 + on-policy 구간 선택적)**: KD를 두 모드로 나눠 운용한다.
- **기본 모드 — 오프라인 teacher-forcing**: 티처의 정답 궤적은 고정이므로 사전 1-pass로 top-k 로짓을 덤프해두고 본 학습에서는 티처를 GPU에 올리지 않는다. 학습량의 대부분을 이 저비용 경로로 처리(VRAM·처리량 이득, `PROJECT.md` §5 시간 단축 레버 1).
- **교정 모드 — on-policy / scheduled sampling 구간**: exposure bias가 누적되는 것을 막기 위해, 학습 후반부(혹은 주기적 구간)에 student 자기 생성 롤아웃을 라이브 티처가 스코어링하는 on-policy 스텝을 **선택적으로** 섞는다. 이 구간에서만 티처가 GPU에 상주(+14GB)하며, 파라미터 미샤딩 기준 GPU당 여유(§`PROJECT.md` §5.3)로 감당 가능. scheduled sampling으로 teacher-forcing→self-rollout 비중을 점진 전환하는 방식이 기본형.
- 순수 teacher-forcing only vs 이 혼합안의 효과 차이는 Phase 5 KD 방식 ablation(`TRAINING_CURRICULUM.md` §2)에서 정량화한다.

### 7.4 침묵/비발화 토큰의 클래스 불균형
대화 대부분이 침묵/청취 상태이고 발화 프레임은 상대적으로 희소. 순수 KD loss(프레임별 KL)를 그대로 쓰면 모델이 "항상 침묵을 예측"하는 쪽으로 쉽게 수렴할 수 있음. 발화/전환 구간에 가중치를 더 주는 등 불균형 처리 필요.

**주의(중복 항 금지)**: 이 불균형 처리는 **semantic KD 내부의 프레임 가중치**로 해결한다. 별도의 "발화/비발화 보조 distill" 항을 만들 필요가 없다 — self/user 역할 구분은 Role Token(`ARCHITECTURE.md` §3·§6.2)이, 발화/침묵 여부는 semantic 토큰 값 자체가 이미 담당하므로, 별도 신호를 추가하면 중복이다.

### 7.5 오디오 코드북 임베딩 드리프트
Moshi 코드북 값으로 초기화한 임베딩이 학습 초반부터 급격히 업데이트되면, KD 타겟(원본 Moshi 로짓)이 가리키는 "의미"와 student의 현재 임베딩 공간이 어긋나기 시작. 초반 몇 스텝은 오디오 임베딩(최소한 코드북 부분)을 freeze하거나 낮은 lr로 warmup한 뒤 풀어주는 전략 고려.

### 7.6 백본의 원래 능력 파괴 (Catastrophic Forgetting)
Qwen3 8B에 오디오 토큰을 얹고 KD+TTS joint loss로 계속 학습하면, 프로젝트 전제인 "Qwen3의 강력한 멀티링구얼 텍스트 능력"이 서서히 퇴화할 위험. 순수 텍스트 loss(소량이라도)를 학습 내내 anchor로 유지 필요. 이 부분이 무너지면 일본어 제로샷 전이 실험(Phase 4) 자체가 무의미해짐.

### 7.7 티처의 오류까지 그대로 전파
원본 Moshi 자체도 완벽한 모델이 아니라 알려진 약점(VAD 오판, 약 5분 컨텍스트 한계 등)이 있음. KD는 기본적으로 티처의 행동을 복제하는 것이므로 티처의 체계적 오류도 그대로 student에 이식됨. 원본 Moshi의 약점 목록을 미리 정리해두고, KD 이후 student가 동일 약점을 보이는지 별도 체크 권장.

### 7.8 스트리밍 프레임 정렬
두 스트림(발화/청취)이 프레임 단위로 동기화되어야 하는데, Qwen3 백본의 시퀀스 처리 방식(캐싱, 포지션 인코딩)이 원본 Moshi의 프레임 레이트/윈도우 가정과 다르면 KD 타겟과 student 예측 시점 간 **temporal misalignment**가 생길 수 있음. 코드 레벨에서 조용히 발생하는 버그이므로, 특정 turn-taking 이벤트 시점에서 두 모델의 예측이 같은 timestep을 가리키는지 별도 검증 필요.

### 7.9 Acoustic graft 시 음색 드리프트 (Phase 3.5)
turn-event 국소 acoustic KD를 얹을 때, teacher의 고정 화자(Moshi-ko/ka) 음색이 클로닝된 목소리로 새어들 위험. 특히 **timbre와 prosody의 학습 속도차**로 인해 "prosody는 아직 안 왔는데 timbre는 이미 샌" 최악 지점이 생길 수 있음. 대응(`ARCHITECTURE.md` §5.3): 페어드 voice-prompt + ko-ka interpolation + 저weight/저LR + turn-event 국소화 + **guard 지표(turn-event 구간 화자 유사도)로 early-stop**. 진단: 일반 발화가 아니라 **barge-in/interruption 프레임의 화자 유사도**를 국소 측정해야 드리프트를 잡음.

### 7.10 voice-prompt와 콘텐츠 결합의 실패 모드
과거 "참조 + learnable VQ 삽입" 시도는 음색은 이동했으나 content가 붕괴(word salad, `ARCHITECTURE.md` §4.4)했음. PersonaPlex가 learnable VQ 없이 프리픽스만으로 달성함을 확인했으므로 그 방식(§5.2)을 따르되, 도입 시 **텍스트 채널을 바꿨을 때 오디오가 따라 바뀌는지(결합 강도)와, voice-prompt가 content 예측을 오염시키지 않는지**를 별도 검증. system prompt 구간 loss 마스킹·non-semantic 다운웨이팅 누락이 흔한 원인일 수 있음.

**우선순위**: §7.3(exposure bias)과 §7.6(catastrophic forgetting)이 논문 성패에 가장 직접적 영향을 줄 것으로 예상. §7.1·7.2는 설계 단계에서 KD 손실 함수를 어디에 걸지 미리 명확히 정의해두면 자연히 해결됨. §7.9는 Phase 3.5에 진입할 때만 활성화되는 조건부 리스크.

## 8. 컨트리뷰션 성립 여부에 대한 메타 검토

**"한국어 멀티턴 전이가 자명한 것 아닌가?"에 대한 답**: 아니다. 텍스트 도메인의 cross-lingual zero-shot transfer(mBERT 기반 DST 등)는 잘 알려진 현상이지만, 이는 "의미/내용"의 전이이다. Haan이 겨냥하는 것은 오디오 채널의 **시간적·구조적 행동(turn-taking)**의 전이이며, 전이 경로도 단순 zero-shot 추론이 아니라 **서로 다른 두 목적함수의 그라디언트를 혼합하는 compositional generalization**이다. 이는 텍스트 문헌과 성격이 다르고, 스피치 full-duplex 도메인에서는 선행 연구가 사실상 없다. 따라서 성공 시 명확한 기여이며, 완전 실패보다 "어디까지 되고 어디서 깨지는가"를 규명하는 부분적 성공/실패 리포트가 논문 스토리로서 더 탄탄할 수 있다.

**실패 시 최소 요구사항**: 위 §1~7 중 어느 지점에서, 왜 막혔는지를 분석해 리포트할 것 (예: "메커니즘은 전이됐지만 언어 식별과 결합된 shortcut 때문에 트리거가 안 됐다"). 이것 자체로 의미 있는 negative result가 된다.
