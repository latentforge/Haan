# Haan 데이터 전략

## 1. 문제 상황

한국어 멀티턴 음성 대화 데이터셋은 수집 난이도가 매우 높다. Haan은 이를 회피하기 위해 **한국어 싱글턴 데이터만으로 멀티턴 능력이 창발하는지**를 검증하는 것이 핵심 가설이다. 그런데 한국어 싱글턴(페어 구조) 데이터조차 충분한 양의 실제 녹음이 부족해, Higgs Audio v2 같은 모델로 assistant 응답 쪽을 **합성**해야 하는 상황이다.

## 2. "싱글턴"의 두 가지 의미 — 반드시 구분할 것

1. **고립된 발화(isolated utterance)**: "이 문장을 한국어로 읽어라"는 순수 TTS 데이터. 입력(context)과 출력(response) 사이에 아무 관계가 없고, 텍스트→음성 매핑만 학습됨.
2. **한 쌍짜리 대화(single-turn dialogue pair)**: (질문/맥락) → (그에 대한 응답) 형태로, 대화가 한 번만 오가지만 "듣고 → 그에 맞게 반응한다"는 조건부 구조는 살아있는 데이터.

### 2.1 순수 고립 발화만 쓸 경우의 근본적 문제
순수 TTS 발화 데이터(1번)는 "한국어를 발음하는 법"만 가르치지, "무언가를 듣고 한국어로 적절히 반응하는 법"은 전혀 가르치지 않는다. 영어 멀티턴 KD와 그라디언트를 섞어도 그건 "언제 말하고 언제 침묵할지"(형식/타이밍)를 알려줄 뿐, "무엇을 말해야 맥락에 맞는지"(내용)에 대한 신호는 부재한 채로 남는다. 이는 제로샷 전이가 아니라 애초에 학습 신호가 존재하지 않는 능력을 기대하는 것에 가깝다.

### 2.2 기본 방침: 합성 싱글턴(페어 구조)을 기본으로 채택
컨트리뷰션이 성립하려면 최소한 "듣고 반응한다"는 구조 자체는 학습 데이터에 있어야, 그 위에서 "멀티턴으로 확장되는가"라는 질문이 의미를 가진다.

**선례**: J-Moshi-ext(멀티스트림 TTS로 생성한 합성 대화 데이터 추가 학습)가 J-Moshi(순수 녹음 대화 데이터만 학습)보다 더 나은 성능을 보였다는 결과가 있어, 합성 TTS 기반 대화 데이터가 실제로 도움이 된다는 것은 경험적으로 확인된 바 있다.

## 3. 합성 파이프라인 설계 (Higgs Audio v2 활용)

### 3.1 2단계 분리 원칙
Higgs Audio v2에게 "질문에 맞는 답을 생성해서 음성으로 만들어라"를 통째로 맡기지 말고:
1. **콘텐츠 생성**: 텍스트 레벨에서 한국어 싱글턴 질문-답변 쌍을 먼저 확보 (기존 한국어 instruction/QA 코퍼스, 혹은 별도 LLM으로 생성)
2. **음성 실현**: 그 텍스트를 Higgs Audio v2 등 TTS로 음성화

이렇게 나누면 "대화 내용의 적절성"과 "음성 합성 품질"이라는 서로 다른 실패 지점을 분리해서 디버깅할 수 있다.

### 3.2 주의사항
- **Synthetic-to-synthetic 오염 위험**: Higgs Audio v2가 생성한 음성으로만 학습하면 Haan이 결국 그 TTS 모델의 운율/화자 스타일/합성 아티팩트를 흡수하게 됨. 화자/음색을 다양화(멀티스피커/보이스클로닝 활용)하고, 품질 낮은 샘플은 ASR 재전사로 필터링하는 절차 필요.
- **화자 조건화 = in-context voice-prompt (`ARCHITECTURE.md` §5.2)**: 음색은 참조 음성을 agent audio 채널에 프리픽스로 까는 방식으로 공급한다. PersonaPlex의 Hybrid System Prompt 기법을 **독립 구현**(가중치·데이터 미사용, 라이선스 회피)한다. 학습 샘플마다 다양한 참조 음성을 페어링해 "프롬프트대로 음색 생성"을 학습(voice-cloning CE).
- **ko-ka interpolation으로 음색 다양화**: 검증된 ko+ka 가중합으로 중간 목소리 연속체를 생성해 참조 음성 풀을 늘린다. 특히 Phase 3.5 acoustic prosody graft에서 "prosody는 음색 불변" 신호를 강화하는 데 사용(`TRAINING_CURRICULUM.md` Phase 3.5, `ARCHITECTURE.md` §5.3). 한계: ko↔ka 1축 연속체이지 전체 voice-space 커버는 아님.
- **User 발화는 가능하면 실제 음성 사용**: 최소한 입력(맥락) 쪽이라도 실제 사람 음성(기존 한국어 음성 코퍼스에서 발췌)으로 구성하면 완전히 합성된 화자끼리의 폐쇄 루프보다 실제 음향 분포에 더 가까워지고, 청취 채널만큼은 진짜 신호를 학습하게 됨.
- **논문 서술상 명확화**: "싱글턴 데이터만으로 성공했다"는 컨트리뷰션 문구를 쓸 때, 그게 "고립 발화" 기준이 아니라 "페어 구조 싱글턴" 기준임을 명확히 밝힐 것. **"대화 쌍 구조는 유지하되 여러 턴에 걸친 문맥 누적/기억이 필요 없는 형태"**로 정의를 좁혀 제시.

## 4. 고립 발화만으로 성공하는 경로 (강한 버전, 컨트리뷰션 확대 옵션)

페어 구조 데이터조차 없이 순수 고립 발화만으로 성공한다면 컨트리뷰션이 훨씬 커진다. 이를 위한 명시적 설계.

### 4.1 핵심 재구성: 능력을 세 개로 분해
"한국어로 대화하는 능력"은 이미 Qwen3 백본 안에 **텍스트 레벨로 존재**한다(Qwen3는 한국어 멀티턴 대화를 텍스트로는 이미 잘함). 문제는 이 능력이 오디오 인터페이스를 통해 입출력될 수 있느냐일 뿐이다. Moshi의 Inner Monologue(자기 발화에 대응하는 텍스트 토큰을 오디오와 시간 정렬해 함께 생성하는 장치)를 활용해 다음과 같이 분리:

| 요소 | 담당 | 필요 데이터 |
|---|---|---|
| 내용/추론 (무엇을 말할지) | Qwen3가 이미 보유 | 텍스트 anchor loss로 보존만 하면 됨, 새 학습 불필요 |
| 음성 실현 (한국어로 어떻게 소리내어 말할지) | 신규 학습 필요 | 고립 발화(텍스트-오디오 쌍) TTS 데이터로 충분 |
| 턴테이킹 (언제 말하고 침묵할지) | 신규 학습 필요 | 영어 멀티턴 KD에서 전이 |

이 셋 중 어느 것도 "한국어 멀티턴 오디오 쌍"을 요구하지 않는다.

### 4.2 남은 구멍: "듣기" 방향 — 같은 데이터의 양방향 재활용
고립 발화 TTS 데이터는 보통 "텍스트→오디오(말하기)" 한 방향만 가르치므로, 사용자 채널로 들어오는 한국어 오디오를 이해하는 능력이 빠진다. 같은 페어 데이터를 두 번 재활용:
- **TTS 방향**: (텍스트 조건) → 오디오 생성 (출력 채널 학습)
- **ASR 방향**: (오디오 입력) → 내부 텍스트(Inner Monologue) 예측 (입력 채널/청취 학습)

추가 데이터 수집 없이 같은 페어에서 양방향 신호를 뽑아낼 수 있다.

**주의 (아키텍처와의 연결, `ARCHITECTURE.md` §2~3, §5.4 참조)**: 이 ASR 방향 학습은 원 Moshi의 self/user 완전 분리 임베딩 구조에서는 **거의 필수**였다 — 유저 채널 슬롯에 실제 콘텐츠를 노출시키는 유일한 방법이었기 때문. 본 트랙에서는 두 가지가 이를 완화한다: (a) **공유 임베딩+Role Token**(§3)으로 표현이 공유되고, (b) **Depth가 user 스트림도 생성**하므로(§5.4) user 채널이 직접 그래디언트를 받는다. 따라서 ASR 방향 재활용은 "필수"에서 **"청취 회로를 한국어 콘텐츠에 노출시키는 보강 수단"**으로 위상이 바뀌지만, 그 노출 요구 자체는 남는다.

### 4.3 턴테이킹 스켈레톤 이식 (Temporal Skeleton Transplantation) — **보류/비채택**
> **결정**: turn-taking(동시성 행동 포함) 전이는 **Moshi semantic-only KD**를 실질 소스로 채택한다(`ARCHITECTURE.md` §5.1, `PROJECT.md` §3.2). 스켈레톤 이식은 (a) content-decoupled 타이밍만 주어 내용-조건 타이밍을 못 배우고, (b) 외부 대화에서 타이밍을 빌려오는 우회이며, (c) Moshi KD가 맞장구·barge-in·오버랩 같은 동시성 행동을 이미 체득한 분포로 더 직접 전이하므로, 본 트랙에서는 채택하지 않는다. 아래 서술은 대안 기록용으로만 남긴다.

실제 영어(혹은 일본어) 멀티턴 대화에서 **타이밍 골격만**(누가 언제 말을 시작/종료하는지, 침묵 길이, 오버랩 위치, 맞장구 슬롯)을 추출해, 그 슬롯에 고립된 한국어 TTS 발화를 갈아 끼우는 합성 학습 샘플을 만드는 방법. 내용은 실제로 서로 안 맞아도 무방(질문과 답이 무관해도 됨) — 목적은 "이 정도 duration이면 이 타이밍에 응답이 시작되고, 이 정도면 침묵 유지, 이런 패턴이면 맞장구" 같은 구조적 관례를 명시적으로 노출시키는 것. 발화 길이가 슬롯과 안 맞으면 time-stretch나 길이 맞는 발화 선택으로 조정.

### 4.4 이 강한 버전의 전제 조건과 잔여 리스크
1. **오디오 생성이 실제로 텍스트 채널에 강하게 조건화되어 있어야 함**: Inner Monologue가 형식적으로 존재해도 학습 중 오디오 스트림이 텍스트 채널과 느슨하게만 결합되어 있다면 콘텐츠(Qwen3가 결정)와 음성(그걸 안 따라감)의 괴리가 생길 수 있음. 텍스트 채널을 다른 내용으로 바꿨을 때 오디오 출력이 따라 바뀌는지 검증하는 결합 강도 실험이 선행되어야 함.
2. **컴포지셔널 제너럴라이제이션 위험은 완전히 제거되지 않음**: "듣고(ASR 방향) → 추론하고(Qwen3 기존 능력) → 말하는(TTS 방향)" 것을 셋 다 따로는 배웠어도, 이 셋을 하나의 자기회귀 시퀀스 안에서 연쇄적으로 수행하는 것은 실제로 연습한 적이 없음. 다만 원래(순수 joint loss로 통짜 창발 기대) 설계보다는 실패 표면이 훨씬 좁음.
3. **내용에 맞는 타이밍까지는 못 배울 수 있음**(예: 복잡한 질문엔 약간 늦게 응답 시작) — 콘텐츠와 타이밍이 분리 학습되었기 때문. 성공하더라도 한계점으로 미리 인정하고 갈 것.

### 4.5 투 트랙 권장
시간·인력이 허락하면 §4(고립 발화, 강한 버전)를 메인으로, 안 되면 §3(합성 페어, 약한 버전)을 fallback으로 두는 투 트랙 설계가 현실적. §4는 데이터 엔지니어링(스켈레톤 이식, 양방향 재활용)과 아키텍처 레벨 검증(텍스트-오디오 결합 강도 확인)이 §3보다 추가로 필요하다.

## 5. 데이터 볼륨 관련 참고 (일본어 전이 실험, `TRAINING_CURRICULUM.md` Phase 4)
J-Moshi는 약 67,000시간의 일본어 데이터(J-CHAT 등)를 사용. Haan의 일본어 전이 실험은 0h/1h/10h/100h 스윕으로 이 대비 데이터 효율을 정량 비교하는 것이 목표.
